
# Configurar Diagnostic settings en el Portal de Azure

Este documento describe cómo configurar las `Diagnostic settings` para un recurso de **Azure SQL Database** (o servidor/managed instance) desde el Portal de Azure, de forma que los logs y métricas se envíen a Log Analytics, Storage o Event Hub para análisis y retención.

**Requisitos previos**
- Permisos: `Owner` o `Monitoring Contributor` sobre el recurso, o permisos equivalentes para crear Diagnostic settings.
- Cuenta de destino: un Workspace de Log Analytics, una cuenta de Azure Storage o un Event Hub/namespace para exportar los datos.

---

## 1. Abrir el recurso en el Portal de Azure
1. Inicia sesión en el Portal de Azure (https://portal.azure.com).
2. Navega hasta el recurso: busca la **Base de datos SQL**, **Servidor SQL** o **Managed Instance** que quieras supervisar.

## 2. Acceder a "Diagnostic settings"
1. En el menú del recurso (columna izquierda), busca y selecciona **Monitoring → Diagnostic settings**.
2. En la página de Diagnostic settings verás la lista de configuraciones existentes (si las hay) y el botón **+ Add diagnostic setting**. Haz clic en él.

## 3. Crear una nueva Diagnostic setting (configuración)
1. **Name**: introduce un nombre descriptivo (ej.: `diag-sql-prod-logs`).
2. **Logs**: marca las categorías de logs que quieras capturar. Para Azure SQL conviene considerar:
   - `SQLInsights` (telemetría y métricas específicas de SQL)
   - `Errors` (errores reportados por el servicio)
   - `Deadlocks` (gráficas de deadlock)
   - `Timeouts` (time outs de consultas)
   - `QueryStoreRuntimeStatistics` (estadísticas de Query Store si están disponibles)
3. **Metrics**: habilita la casilla para enviar métricas (si no está marcada por defecto).

### Observación sobre categorías
- Las categorías disponibles pueden variar según el tipo de recurso (Single DB, Managed Instance, SQL Server on VM). Selecciona las categorías que mejor sirvan a tus necesidades de diagnóstico y coste.

## 4. Elegir el destino (s)
Selecciona uno o varios destinos para enviar logs y métricas:

- **Send to Log Analytics workspace**
  - Selecciona el Workspace de Log Analytics donde se almacenarán los logs.
  - Ventaja: consultas KQL, alertas avanzadas, Workbooks y correlación con otras fuentes.

- **Archive to a storage account**
  - Selecciona la cuenta de Storage y el contenedor donde archivar los logs en formato JSON/AVRO.
  - Ventaja: retención a largo plazo (económico) y cumplimiento.

- **Stream to an Event Hub**
  - Permite enviar eventos a consumidores externos o SIEM.

Marca los destinos deseados y configura sus parámetros (por ejemplo Workspace, Storage account, Event Hub namespace).

## 5. Retención y costes
- **Log Analytics retention:** si envías a Log Analytics, la retención por defecto puede ser limitada; ajusta la retención en el Workspace según políticas de coste y cumplimiento.
- **Storage retention:** al archivar en Storage controlas la retención mediante políticas de ciclo de vida del Blob.
- **Coste:** exportar muchas categorías o métricas frecuentemente incrementa costo en Log Analytics y Event Hub. Filtra categorías innecesarias y usa almacenamiento como archivo para retención masiva.

## 6. Guardar la configuración
1. Revisa las opciones y pulsa **Save**.
2. La nueva diagnostic setting se mostrará en la lista y comenzará a enviar datos al destino seleccionado.

## 7. Verificar y consultar los datos
- **En Log Analytics (Kusto):** abre el Workspace y usa la sección `Logs` para consultar con KQL. Ejemplo básico para ver entradas de SQLInsights:

```kql
AzureDiagnostics
| where ResourceProvider == "MICROSOFT.SQL" and Category == "SQLInsights"
| sort by TimeGenerated desc
| take 50
```

- **En Storage:** navega al contenedor configurado, descarga los blobs y procesa los JSON para análisis batch.
- **En Event Hub:** confirma que el consumer está recibiendo eventos (p. ej. un servicio SIEM o una función custom).

## 8. Buenas prácticas y recomendaciones
- **Filtrar categorías**: evita enviar categorías de logs que no necesites para reducir coste.
- **Rutas múltiples**: envía a Log Analytics para análisis y a Storage para archivo a largo plazo.
- **Automatizar**: crear Diagnostic settings mediante ARM/Bicep/Terraform para entornos reproducibles.
- **Seguridad:** restringe el acceso a la cuenta de Storage y al Workspace; vigila claves de Event Hub.
- **Monitorizar el tamaño**: revisa el volumen de datos y ajusta frecuencias/retención.

## 9. Ejemplo ARM (breve) — crear Diagnostic setting hacia Log Analytics
```json
{
  "type": "Microsoft.Insights/diagnosticSettings",
  "apiVersion": "2017-05-01-preview",
  "name": "diag-sql-prod-logs",
  "properties": {
    "workspaceId": "/subscriptions/<sub>/resourceGroups/<rg>/providers/Microsoft.OperationalInsights/workspaces/<workspace>",
    "metrics": [ { "timeGrain": "PT1M", "enabled": true } ],
    "logs": [ { "category": "SQLInsights", "enabled": true }, { "category": "Errors", "enabled": true } ]
  }
}
```
