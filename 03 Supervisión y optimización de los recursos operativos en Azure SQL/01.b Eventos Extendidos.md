
# Ejemplo de Capturar inicios de sesión con Extended Events 
# (sesión a nivel de base de datos)

## Crear la sesión
```sql
-- Crea una sesión de eventos denominada 'CaptureLogins' a nivel de base de datos.
-- Captura el evento `sqlserver.login` y registra acciones útiles: nombre de app cliente,
-- nombre del host cliente, nombre de la base de datos y nombre de usuario.
CREATE EVENT SESSION [CaptureLogins] ON DATABASE
ADD EVENT sqlserver.login(
		ACTION(
				sqlserver.client_app_name,
				sqlserver.client_hostname,
				sqlserver.database_name,
				sqlserver.username
		)
)
-- Usa el target 'ring_buffer' (almacenamiento en memoria como XML) y limita memoria.
ADD TARGET package0.ring_buffer
WITH (MAX_MEMORY=4096 KB, EVENT_RETENTION_MODE=ALLOW_SINGLE_EVENT_LOSS);
GO
```

Este bloque crea la sesión. Observaciones importantes:
- `ring_buffer` mantiene los eventos en memoria como XML; es volátil y susceptible a pérdida si el servidor se reinicia o si la cola se sobrescribe.
- `MAX_MEMORY` controla la memoria que el target puede usar; ajústalo según carga y pruebas.
- En entornos IaaS/Managed Instance es preferible usar `event_file` para persistencia en disco; en Azure SQL PaaS el acceso a archivos puede estar restringido, por lo que `ring_buffer` o las `Diagnostic Settings` son las alternativas.
- Filtrar eventos (predicates) es recomendable en producción para reducir volumen (ej.: añadir WHERE para client_app_name o username).

## Iniciar la sesión
```sql
-- Inicia la sesión para comenzar a capturar eventos.
ALTER EVENT SESSION [CaptureLogins] ON DATABASE STATE = START;
GO
```

Iniciar la sesión activa la captura en tiempo real. Requisitos y notas:
- Se necesitan permisos adecuados en la base de datos (por ejemplo `db_owner` o permisos equivalentes) para crear/arrancar sesiones a nivel de base de datos.
- Verifica el impacto en rendimiento: aunque Extended Events es ligero, capturar muchos eventos sin filtros puede aumentar la carga.

## Detener la sesión
```sql
-- Detiene la sesión sin eliminarla. Útil para pausar captura mientras se revisa la salida.
ALTER EVENT SESSION [CaptureLogins] ON DATABASE STATE = STOP;
GO
```

Detener la sesión suspende la captura pero mantiene la definición para reanudarla más tarde.

## Eliminar la sesión
```sql
-- Elimina la sesión de eventos. Útil para limpiar después de pruebas o cuando ya no se necesita.
DROP EVENT SESSION [CaptureLogins] ON DATABASE;
GO
```

Eliminar la sesión borra su definición y cualquier dato asociado en memoria. Asegúrate de exportar o consultar los datos antes de eliminarla si quieres conservar la información capturada.

## Inspeccionar el target (XML raw)
```sql
-- Consulta básica que devuelve el contenido XML del target asociado a la sesión.
SELECT CAST(target_data AS XML) AS target_data
FROM sys.dm_xe_database_session_targets t
JOIN sys.dm_xe_database_sessions s
		ON t.event_session_address = s.address
WHERE s.name = 'CaptureLogins';
```

Este SELECT devuelve el XML crudo almacenado por el `ring_buffer`. Es útil para inspección rápida o depuración,
pero para análisis y reportes conviene extraer campos concretos con `nodes()` y `value()` (ver siguiente comando).

## Parsear el XML del ring_buffer y obtener columnas legibles
```sql
-- Extrae los eventos desde el ring_buffer y desglosa los campos de interés en columnas.
SELECT
		-- Convierte la marca de tiempo UTC del evento a hora local del servidor de base de datos
		DATEADD(MINUTE, DATEDIFF(MINUTE, GETUTCDATE(), SYSDATETIME()),
				split.xed.value('(@timestamp)[1]', 'datetime2(7)')
		) AS event_time_local,
		split.xed.value('(@name)[1]', 'nvarchar(100)') AS event_name,
		split.xed.value('(action[@name="username"]/value/text())[1]', 'nvarchar(256)') AS username,
		split.xed.value('(action[@name="client_app_name"]/value/text())[1]', 'nvarchar(256)') AS client_app_name,
		split.xed.value('(action[@name="client_hostname"]/value/text())[1]', 'nvarchar(256)') AS client_hostname,
		split.xed.value('(action[@name="database_name"]/value/text())[1]', 'nvarchar(256)') AS database_name
FROM
(
		-- Selecciona el XML del target ring_buffer para la sesión especificada
		SELECT CAST(t.target_data AS XML) AS target_data
		FROM sys.dm_xe_database_session_targets t
		INNER JOIN sys.dm_xe_database_sessions s
				ON t.event_session_address = s.address
		WHERE s.name = 'CaptureLogins'
			AND t.target_name = 'ring_buffer'
) AS tab
-- Despliega cada nodo <event> del RingBufferTarget/event
CROSS APPLY tab.target_data.nodes('/RingBufferTarget/event') AS split(xed)
ORDER BY event_time_local DESC;
GO
```

Explicación del SELECT de parseo:
- Se hace `CROSS APPLY` con `nodes()` para convertir cada nodo `<event>` del XML en una fila relacional.
- `value()` extrae atributos y elementos (ej. `@timestamp`, `@name`) y las `action` asociadas.
- La expresión `DATEADD(...)` convierte la marca de tiempo UTC del evento a la hora local del servidor; ajustar según zona si es necesario.

## Variantes y recomendaciones
- Persistencia: para capturas persistentes en IaaS/Managed Instance usar `ADD TARGET package0.event_file (SET filename=N'...')` y rotación de archivos.
	```sql
	-- Variante: usar event_file (requiere acceso a sistema de archivos; útil en VM o Managed Instance)
	ADD TARGET package0.event_file (SET filename=N'C:\XE\CaptureLogins.xel');
	```
- Filtrado: añadir predicates para reducir eventos capturados, p. ej. `WHERE (sqlserver.username <> 'sa')` o filtrar por `client_app_name`.
	```sql
	-- Ejemplo de evento con filtro para aplicaciones que empiezan por 'MyApp'
	ADD EVENT sqlserver.login(WHERE (sqlserver.client_app_name LIKE 'MyApp%'))
	```
- Limpieza: eliminar la sesión cuando ya no se necesita.
	```sql
	-- Elimina la sesión
	DROP EVENT SESSION [CaptureLogins] ON DATABASE;
	```

## Notas específicas para Azure SQL (PaaS) y Managed Instance
- En **Azure SQL Database (Single/Elastic)** algunas capacidades de Extended Events están disponibles a nivel de base de datos, pero el uso de `event_file` puede estar restringido; `ring_buffer` y la exportación mediante `Diagnostic Settings` (Log Analytics / Storage / Event Hub) son las opciones recomendadas para conservación y análisis a largo plazo.
- En **Managed Instance** hay mayor paridad con SQL Server on-prem y es factible usar `event_file` y otras opciones más completas.
- Para análisis a escala y retención mayor a 93 días, exporta datos de diagnóstico a **Azure Storage** o **Log Analytics** y consulta con KQL.
