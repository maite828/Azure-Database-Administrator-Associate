
# Eventos Extendidos — guardar en Azure Blob Storage (ejemplo)

Este documento replica el ejemplo de `01.b Eventos Extendidos.md` pero configurando el target `event_file`
para escribir los ficheros `.xel` directamente en un contenedor de Azure Blob Storage.

IMPORTANTE: la capacidad de escribir directamente en Blob Storage desde Extended Events depende del tipo de despliegue:
- **Managed Instance** y **SQL Server en VM (IaaS)**: soporte completo para `event_file` hacia ubicaciones externas.
- **Azure SQL Database (Single/Elastic)**: el uso directo de `event_file` hacia Blob puede estar restringido; en PaaS se recomienda usar `Diagnostic Settings` para exportar telemetría a Storage o Log Analytics.

---

## Preparar clave maestra y credenciales para Azure Blob
```sql
-- 1) Crear Database Master Key si no existe (necesaria para credenciales en la BD)
IF NOT EXISTS (SELECT * FROM sys.symmetric_keys WHERE name = '##MS_DatabaseMasterKey##')
BEGIN
    CREATE MASTER KEY ENCRYPTION BY PASSWORD = 'Strong!Passw0rdChangeMe';
END
GO

-- 2) Crear una Database Scoped Credential que apunte al contenedor de Blob usando SAS
-- Reemplaza los valores <storage-account>, <container> y <SAS_TOKEN> por tus datos.
-- Nota: para SAS tokens, omite el carácter inicial '?' si la plataforma lo requiere, o inclúyelo según el caso.
-- Ejemplo: SECRET = 'sv=2024-...&ss=bfqt&...'

IF NOT EXISTS (SELECT * FROM sys.database_scoped_credentials WHERE name = 'BlobStorageCredential')
BEGIN
    CREATE DATABASE SCOPED CREDENTIAL BlobStorageCredential
    WITH IDENTITY = 'SHARED ACCESS SIGNATURE',
         SECRET = '<SAS_TOKEN_without_leading_question_mark>' ;
END
ELSE
BEGIN
    -- Si ya existe, actualiza el secreto (SAS) con ALTER
    ALTER DATABASE SCOPED CREDENTIAL BlobStorageCredential
    WITH SECRET = '<SAS_TOKEN_without_leading_question_mark>' ;
END
GO
```

Notas sobre seguridad y permisos:
- El SAS token debe tener permisos de escritura/creación (`w` y `c` o `sp`) sobre el contenedor donde se escribirán los ficheros.
- Guarda los SAS tokens en un Key Vault o sistema seguro; evita hardcodearlos en scripts productivos.

## Crear la sesión Extended Events con target `event_file` apuntando a Blob
```sql
-- Crea la sesión y escribe los archivos .xel en Azure Blob Storage
-- Reemplaza <storage-account> y <container> por los reales.
CREATE EVENT SESSION [CaptureLoginsToBlob] ON DATABASE
ADD EVENT sqlserver.login(
    ACTION(
        sqlserver.client_app_name,
        sqlserver.client_hostname,
        sqlserver.database_name,
        sqlserver.username
    )
)
ADD TARGET package0.event_file
(
    SET filename = 'https://<storage-account>.blob.core.windows.net/<container>/CaptureLogins.xel',
        max_file_size = 5242880, -- tamaño máximo por archivo en bytes (5 MB)
        max_rollover_files = 10
);
GO
```

Explicaciones:
- `filename` indica la URL completa del blob donde se escribirán los eventos. Asegúrate de que el contenedor exista y que la credencial SAS tenga permisos.
- `max_file_size` y `max_rollover_files` controlan la rotación de ficheros para no saturar una sola unidad.

## Iniciar la sesión
```sql
ALTER EVENT SESSION [CaptureLoginsToBlob] ON DATABASE STATE = START;
GO
```

## Detener la sesión (pausar la captura)
```sql
ALTER EVENT SESSION [CaptureLoginsToBlob] ON DATABASE STATE = STOP;
GO
```

## Eliminar la sesión (limpieza)
```sql
DROP EVENT SESSION [CaptureLoginsToBlob] ON DATABASE;
GO
```

## Leer ficheros .xel desde Azure Blob (opciones)
- En Managed Instance o VM puedes descargar los .xel y usar `sys.fn_xe_file_target_read_file` para leerlos en T-SQL.
- Si los ficheros están en Blob Storage, descarga localmente o usa herramientas externas (Azure Storage Explorer, AzCopy) para obtenerlos y analizarlos.

Ejemplo (leer ficheros descargados en un servidor con SQL Server):
```sql
-- Leer ficheros .xel con la función de lectura de XE (ejecutar en una instancia que tenga acceso al fichero)
SELECT event_data = CAST(event_data AS XML)
FROM sys.fn_xe_file_target_read_file('C:\XE\CaptureLogins*.xel', NULL, NULL, NULL);
```

## Actualizar la credencial cuando cambia el SAS
```sql
-- Si el SAS expira y obtienes uno nuevo, actualiza la credencial con ALTER.
ALTER DATABASE SCOPED CREDENTIAL BlobStorageCredential
WITH SECRET = '<NEW_SAS_TOKEN_without_leading_question_mark>' ;
GO

-- Alternativamente, puedes eliminar y crear de nuevo la credencial:
DROP DATABASE SCOPED CREDENTIAL BlobStorageCredential;
GO
CREATE DATABASE SCOPED CREDENTIAL BlobStorageCredential
WITH IDENTITY = 'SHARED ACCESS SIGNATURE',
     SECRET = '<NEW_SAS_TOKEN_without_leading_question_mark>' ;
GO
```

## Consideraciones y variantes
- En entornos productivos prefiera Managed Identity + Key Vault para obtener SAS dinámico y no almacenar secretos en la base de datos.
- Si no es posible escribir directamente a Blob desde la base de datos (restricción PaaS), configure `Diagnostic Settings` para enviar Extended Events/telemetría a Storage o Log Analytics.
- Controla la rotación y limpieza de blobs con políticas de ciclo de vida del Storage.
