# Ejemplo de seguridad de fila en SQL Server con Northwind

## 1. Objetivo del ejemplo

La finalidad es hacer que cada usuario solo pueda ver las filas de la tabla Orders que le correspondan.

En este ejemplo:

- El usuario ventas_usa verá únicamente los pedidos de clientes de USA.
- El usuario ventas_uk verá únicamente los pedidos de clientes de UK.
- Un usuario administrador podrá ver todos los pedidos.

Para conseguirlo, usaremos:

- Una función inline table-valued
- Una Security Policy
- Usuarios de prueba
- Consultas para validar el comportamiento

## 2. Concepto de Row-Level Security

La seguridad de fila en SQL Server permite restringir qué filas puede ver o modificar un usuario, sin tener que cambiar las consultas de la aplicación.

Es decir, un usuario puede ejecutar:

SELECT * FROM Orders;

y SQL Server devolverá solo las filas que tenga permitidas según la política de seguridad.

## 3. Preparación del escenario

En Northwind, la tabla Orders no guarda directamente el país en cada pedido, pero sí tiene el CustomerID.
La tabla Customers contiene el campo Country.

Por tanto, la lógica será:

- Si el usuario es ventas_usa, podrá ver pedidos de clientes cuyo país sea USA.
- Si el usuario es ventas_uk, podrá ver pedidos de clientes cuyo país sea UK.
- Si el usuario es dbo o un usuario administrador, verá todos.

## 9. Crear la función de predicado de seguridad

Esta función será la pieza central de la seguridad de fila.

```sql
CREATE FUNCTION Security.fn_RLS_OrdersByCountry(@CustomerID NCHAR(5))
RETURNS TABLE
WITH SCHEMABINDING
AS
RETURN
(
	SELECT 1 AS fn_RLS_OrdersByCountry_Result
	WHERE
		-- El supervisor puede ver todos los pedidos
		USER_NAME() = 'supervisor'

		OR

		-- El usuario ventas_usa solo ve pedidos de clientes de USA
		(
			USER_NAME() = 'ventas_usa'
			AND EXISTS
			(
				SELECT 1
				FROM dbo.Customers c
				WHERE c.CustomerID = @CustomerID
				  AND c.Country = 'USA'
			)
		)

		OR

		-- El usuario ventas_uk solo ve pedidos de clientes de UK
		(
			USER_NAME() = 'ventas_uk'
			AND EXISTS
			(
				SELECT 1
				FROM dbo.Customers c
				WHERE c.CustomerID = @CustomerID
				  AND c.Country = 'UK'
			)
		)

		OR

		-- El propietario dbo también puede ver todos los pedidos
		USER_NAME() = 'dbo'
);
GO
```

Explicación

¿Qué hace esta función?

Recibe el `CustomerID` de cada fila de `Orders` y devuelve una fila solo cuando el usuario tiene permiso para verla.

Partes importantes

- `RETURNS TABLE`: una función de predicado de RLS debe ser una función tabular inline.
- `WITH SCHEMABINDING`: obliga a que los objetos usados por la función no se modifiquen de forma que rompan su definición.
- `USER_NAME()`: identifica el usuario actual de la sesión.
- `EXISTS (...)`: comprueba si el pedido pertenece a un cliente del país autorizado.

Lógica aplicada

- `supervisor` ve todo
- `ventas_usa` solo pedidos de clientes de USA
- `ventas_uk` solo pedidos de clientes de UK
- `dbo` ve todo

## 10. Crear la Security Policy

Ahora asociamos la función con la tabla Orders.

```sql
CREATE SECURITY POLICY Security.OrdersFilterPolicy
ADD FILTER PREDICATE Security.fn_RLS_OrdersByCountry(CustomerID)
ON dbo.Orders
WITH (STATE = ON);
GO
```

Explicación

`CREATE SECURITY POLICY` crea la política de seguridad.

`ADD FILTER PREDICATE` indica que se aplicará un filtro automático.

`Security.fn_RLS_OrdersByCountry(CustomerID)` llama a la función pasando el `CustomerID` de cada fila de `Orders`.

`WITH (STATE = ON)` activa la política inmediatamente.

Efecto

A partir de este momento, cualquier `SELECT` sobre `dbo.Orders` quedará filtrado según el usuario.

## 15. Validar el país de los pedidos devueltos

```sql
EXECUTE AS USER = 'ventas_usa';
GO

SELECT o.OrderID,
	   o.CustomerID,
	   c.CompanyName,
	   c.Country,
	   o.OrderDate
FROM dbo.Orders o
INNER JOIN dbo.Customers c
	ON o.CustomerID = c.CustomerID
ORDER BY o.OrderID;
GO

REVERT;
GO
```

Explicación

Esta consulta permite verificar visualmente que el usuario `ventas_usa` solo ve clientes con `Country = 'USA'`.

## 16. Ejemplo de cómo funciona “sin cambiar la consulta”

Una de las ventajas de RLS es que la aplicación no necesita conocer la lógica de seguridad.

La misma consulta:

```sql
SELECT OrderID, CustomerID, OrderDate
FROM dbo.Orders;
GO
```

devolverá resultados distintos según el usuario que la ejecute.

Explicación

La seguridad no está en la consulta, sino en la política definida en SQL Server.

## 17. Consultar las políticas de seguridad creadas

```sql
-- Ver las políticas de seguridad existentes en la base de datos
SELECT name, is_enabled
FROM sys.security_policies;
GO

-- Ver los predicados asociados
SELECT sp.name AS PolicyName,
	   spp.type_desc,
	   OBJECT_NAME(spp.target_object_id) AS TargetTable,
	   OBJECT_NAME(spp.predicate_id) AS PredicateFunction
FROM sys.security_policies sp
INNER JOIN sys.security_predicates spp
	ON sp.object_id = spp.object_id;
GO
```

Explicación

Estas vistas de catálogo permiten revisar:

- qué políticas existen
- si están activas
- sobre qué tabla aplican
- qué función utilizan

## 18. Desactivar temporalmente la política

Si quieres hacer pruebas sin el filtro:

```sql
ALTER SECURITY POLICY Security.OrdersFilterPolicy
WITH (STATE = OFF);
GO
```

Explicación

La política sigue existiendo, pero deja de aplicarse.

Es útil para mantenimiento o validaciones.

Para volver a activarla:

```sql
ALTER SECURITY POLICY Security.OrdersFilterPolicy
WITH (STATE = ON);
GO
```

CREATE SCHEMA Security;
GO

CREATE FUNCTION Security.fn_RLS_OrdersByCountry(@CustomerID NCHAR(5))
RETURNS TABLE
WITH SCHEMABINDING
AS
RETURN
(
	SELECT 1 AS fn_RLS_OrdersByCountry_Result
	WHERE
		USER_NAME() = 'supervisor'
		OR
		(
			USER_NAME() = 'ventas_usa'
			AND EXISTS

## 21. Conclusiones

Con este ejemplo has implementado seguridad de fila en SQL Server sobre la tabla `Orders` de Northwind.

Lo más importante del ejemplo

- La tabla protegida es `dbo.Orders`
- La restricción se basa en el país del cliente relacionado en `dbo.Customers`
- La seguridad se implementa con:
  - una función de predicado
  - una `SECURITY POLICY`
- Las consultas no cambian; el filtrado lo hace SQL Server automáticamente

Ventajas

- Centraliza la seguridad en la base de datos
- Reduce errores en la aplicación
- Hace más sencillo controlar qué filas puede ver cada usuario

Limitación del ejemplo

Este caso está simplificado para fines didácticos. En un entorno real, normalmente se usaría:

- una tabla de asignación de usuarios y permisos
- roles corporativos
- integración con usuarios reales de Active Directory o SQL Logins


> Este es un ejemplo realizado con la base de datos Northwind para mostrar cómo implementar seguridad de fila (Row-Level Security) en SQL Server sobre la tabla Orders, filtrando automáticamente los pedidos visibles según el usuario que ejecuta la consulta.
